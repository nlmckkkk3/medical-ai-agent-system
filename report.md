# MedChoice — 医疗健康智能助手

## 1. 项目概述

MedChoice 是一款面向普通消费者的医疗健康智能助手。它帮助用户在面对体检套餐选择、药品/保健品对比、体检报告解读、就医科室推荐四个场景时，做出信息充分、有理有据的消费决策。

与传统的"套壳 Chatbot"不同，MedChoice 内部包含 4 个协作 Agent（安全检测、意图路由、场景业务、反思校验），通过完整的智能体工作流——安全检测、意图路由、信息补全、RAG 检索、对比分析、反思修正、人机协同——完成任务闭环。系统支持流式输出（实时显示生成内容）、关键词匹配 + BGE 语义嵌入双模式检索（37 篇中文医疗知识文档）、多轮对话记忆和深色毛玻璃 CSS 主题界面。用户每次使用都会沉淀健康画像和偏好数据，形成"越用越懂你"的复利效应。

项目使用 Python 开发，以 Claude Code 作为 AI 编程协作工具完成需求分析、架构设计、代码生成、调试测试和报告撰写的全过程。

## 2. 问题背景与目标用户

### 2.1 问题背景

根据人民网 2026 年全国两会调查数据，"AI+医养"是超过 60% 网民期待的数字应用场景。同时，量子位智库《2026 年中国 AI 应用全景图谱报告》显示，医疗健康是 AI 渗透率第二高的消费决策领域（47%），仅次于旅游出行（50%）。

然而，目前市场上的健康类 AI 工具大多只是简单的单轮问答，缺少结构化对比、真实知识库支撑和安全边界。用户面对体检、买药、看报告这些高频场景时，仍然需要自己花大量时间搜网页、问朋友、挂号咨询。一个能真正"做事"而非仅仅"聊天"的医疗健康智能助手，是真实存在的市场空白。

### 2.2 目标用户

- **核心用户**：25-45 岁城市居民，每年有体检习惯，有健康消费需求（买药、保健品、体检）
- **延伸用户**：需要为父母/子女做健康消费决策的家庭成员
- **场景特征**：不是紧急就医，而是"消费前做功课"——不知道选哪个体检套餐、不知道两种药差在哪、看不懂体检报告上的异常指标

### 2.3 核心价值

| 传统方式 | MedChoice |
|---------|-----------|
| 打开多个网页逐个查套餐、比价格 | 输入画像，一张对比表秒出结果 |
| 药品说明书看不懂，问朋友不靠谱 | 拍照上传，结构化对比 + 标注"请咨询医生" |
| 体检报告一堆箭头，不知道怎么挂号 | 解析指标 → 解释含义 → 建议科室，标注"仅供参考" |
| 身体不适不知道该挂哪个科 | 描述症状 → 智能匹配科室 → 就诊建议，标注"仅供参考" |

## 3. 智能体系统设计

### 3.1 多智能体角色分工

系统内部包含 4 个具备独立职责的 Agent，形成多智能体协作体系：

| Agent 角色 | 职责 | 输入 | 输出 |
|-----------|------|------|------|
| **安全检测 Agent** | 拦截诊断请求、过滤危险内容、边界约束 | 用户原始输入 | 通过 / 拦截 + 引导语 |
| **意图路由 Agent** | 识别用户意图，分发到对应业务场景 | 通过安检的输入 | 场景标签 + 路由指令 |
| **场景业务 Agent** | 执行具体场景的完整工作流（信息补全→检索→对比→生成） | 场景标签 + 用户输入 | 结构化建议/对比表/解读报告 |
| **反思校验 Agent** | 多维度内容合规与事实校验 | 场景 Agent 的输出 | 通过 / 打回修正指令 |

协作流程：安全检测 → 意图路由 → 场景业务 → 反思校验 → 用户确认。四个 Agent 串行协作，各司其职，反思 Agent 不通过则打回场景 Agent 重新生成（最多重试 2 次）。

### 3.3 四大业务场景

#### 场景 1：体检套餐选择

用户输入个人画像（年龄、性别、职业、预算、家族病史等），智能体从知识库检索匹配套餐，结构化对比后推荐最优选择。

**示例输入**："我 35 岁男程序员，预算 1500，选哪个体检套餐？"

**完整工作流**：

```
用户输入 → 安全检测(通过) → 意图路由(体检套餐选择)
    → 信息补全(追问：家族病史？近期不适？已有慢性病？)
    → RAG检索(从知识库检索匹配套餐)
        ├─ 正常：获得2-3个候选套餐
        └─ 异常：检索为空 → 降级告知，建议调整预算
    → 对比分析(提取对比维度：项目数/核心器官/肿瘤标志物/价格/适用人群)
        └─ 异常：维度不全 → 展示已有数据，缺失标注"暂无数据"
    → 反思校验(边界/一致性/安全性/免责/完整性检查)
        └─ 不通过 → 打回修正，最多2次
    → 输出(对比表+推荐理由+免责声明)
    → 用户确认/修改/追问
```

#### 场景 2：药品/保健品对比

用户上传两个药品说明书或输入药名，智能体解析后从适应症、禁忌、不良反应、价格、医保覆盖等维度生成对比表。

**示例输入**：上传布洛芬和对乙酰氨基酚的说明书图片

**完整工作流**：

```
用户上传文件 → 安全检测(通过) → 意图路由(药品对比)
    → 文档解析(提取关键字段：适应症/禁忌/不良反应/用法用量/价格)
        ├─ 成功：获得结构化字段
        └─ 异常：文件损坏/格式不支持 → 提示支持格式，请用户重传
    → 维度对齐(自动匹配两种药品的可比维度)
    → RAG检索(补充药品知识库信息)
        └─ 异常：检索超时 → 仅用解析结果+标注
    → 对比表生成
    → 反思校验(逐项检查)
    → 输出(对比表+使用建议+免责声明)
    → 用户确认
```

#### 场景 3：体检报告解读

用户上传体检报告 PDF 或图片，智能体解析异常指标，解释可能的含义，建议就诊科室。

**示例输入**：上传体检报告，问"谷丙转氨酶 65 这个箭头什么意思？要去看吗？"

**完整工作流**：

```
用户上传报告 → 安全检测(通过) → 意图路由(报告解读)
    → 文档解析(提取异常指标：名称/数值/参考范围/偏离方向)
    → RAG检索(检索每个异常指标的医学解释)
    → 关联分析(多个异常指标之间的关联)
    → 科室推荐(根据异常指标匹配就诊科室)
    → 反思校验(重点检查：是否越界给出诊断结论)
        └─ 若含诊断表述 → 自动剥离，替换为"建议就医咨询"
    → 输出(指标解读+关联分析+科室建议+免责声明)
    → 用户确认
```

#### 场景 4：就医科室推荐

用户描述症状或不适，智能体从科室知识库检索后推荐合适的就诊科室和就诊前准备建议。

**示例输入**："头痛胸闷应该挂什么科？"

**完整工作流**：

```
用户输入 → 安全检测(通过) → 意图路由(就医科室推荐)
    → RAG检索(从科室知识库检索匹配的科室信息)
    → 症状分析(可能涉及的系统：神经/心血管/呼吸等)
    → 科室推荐(推荐1-3个优先就诊科室，说明每个科室的诊治范围)
    → 就诊建议(就诊前准备、需要带的检查资料等)
    → 反思校验(重点检查：是否越界给出诊断结论)
    → 输出(症状分析+科室推荐+就诊建议+免责声明)
    → 用户确认
```

### 3.4 状态与记忆管理

| 记忆类型 | 存储内容 | 生命周期 | 用途 |
|---------|---------|---------|------|
| **用户画像** | 年龄、性别、职业、预算偏好、家族病史、已有慢性病 | 持久化（文件存储） | 跨会话复用，下次提问无需重复输入 |

| **知识库** | 体检套餐数据、药品说明书、医学科普知识 | 持久化（向量数据库） | 跨用户共享的医学知识资产 |

| 层面 | 沉淀内容 | 复利效果 |
|------|---------|---------|

## 4. 关键设计模式应用

本系统共应用了 7 种课程设计模式 + 多智能体协作机制，不低于课程要求的 4 种：

| 设计模式 | 本系统对应实现 | 归类 |
|---------|---------------|------|
| **路由 (Routing)** | 意图路由 Agent 根据用户输入自动分发到体检选择/药品对比/报告解读/科室推荐四个业务场景，避免单一流程处理所有问题 | 智能体设计模式 |
| **任务规划 (Planning)** | 每个场景内按"信息补全→RAG检索→对比/分析→生成输出"分步执行，确保每步有明确的前置条件 | 智能体设计模式 |
| **工具使用 (Tool Use)** | Agent 主动调用 3 类外部工具：知识库检索工具（RAG）→ 文档解析与结构化提取工具 → 模型调用与计算对比工具 | 智能体设计模式 |
| **反思自省 (Reflection)** | 反思校验 Agent 在每次输出前自动进行 5 维检查（边界/一致性/安全性/免责/完整性），不通过则打回修正 | 智能体设计模式 |
| **记忆机制 (Memory)** | 用户画像持久化（跨会话记忆）+ 会话上下文管理（短时记忆），支撑个性化推荐 | 工程机制 |
| **安全护栏 (Guardrails)** | 安全检测 Agent 前置拦截诊断请求、过滤危险内容、每条输出强制带免责声明 | 工程机制 |
| **人机协同 (Human-in-the-loop)** | 所有输出展示后等待用户确认/修改/追问，关键决策点不自动执行，用户保有最终决定权 | 工程机制 |
| **多智能体协作** | 4 个独立 Agent（安全检测+意图路由+场景业务+反思校验）串行协作，各司其职，符合报告对多智能体协作的考察要求 | 智能体设计模式 |

### 5.1 技术选型

| 组件 | 技术选择 | 选型理由 |

### 5.4 异常处理设计

| 层级 | 异常类型 | 处理策略 |
|------|---------|---------|
| **安全检测 Agent** | 用户输入诊断请求（"我头痛是什么病"） | 拦截 + 引导："我无法提供诊断，建议您前往医院就诊。我可以帮您查询与头痛相关的科室信息" |
| **安全检测 Agent** | 用户输入含危险内容（自伤/伤他） | 拦截 + 提示就医，终止流程 |
| **意图路由 Agent** | 无法识别意图 | 不丢弃 → 返回场景选择引导："您是想了解体检套餐、比较药品还是解读报告？" |
| **意图路由 Agent** | 输入格式异常（乱码/空输入） | 提示用户重新输入 |
| **信息补全阶段** | 用户画像信息严重缺失 | 列出缺失项（年龄/性别/预算等），请用户补充后继续 |
| **RAG 检索** | 检索结果为空 | 降级告知："当前知识库暂无匹配结果，您可以尝试放宽条件" |
| **RAG 检索** | 检索超时 | 降级为 LLM 通用知识 + 明确标注"该信息非知识库来源，请核实" |
| **文档解析** | 文件格式不支持/文件损坏 | 提示支持的文件格式（PDF/JPG/PNG），请用户重新上传 |
| **对比分析** | 对比维度不全（如某药缺少价格数据） | 仅展示有数据的维度，缺失项标注"暂无数据" |
| **LLM 调用** | API 超时 / 频率限制 | 自动重试 1 次，仍失败则返回"服务繁忙，请稍后再试" |
| **反思校验 Agent** | 输出检查不通过 | 打回场景 Agent 修正，最多重试 2 次；仍不通过则降级输出 + 标注"AI 生成内容可能存在误差，建议人工核实" |

### 5.5 安全边界设计

| 安全层级 | 机制 | 说明 |
|---------|------|------|
| **输入过滤** | 安全检测 Agent 前置拦截 | 关键词匹配 + LLM 意图识别双重检测，拒绝诊断/治疗/自伤相关请求 |
| **输出约束** | 反思校验 Agent 边界检查 | 自动剥离诊断结论，替换为就医建议；检查无依据推断并去除 |
| **强制免责** | 每条输出后缀 | 所有输出以"以上信息基于公开资料生成，仅供参考，请结合医生建议做最终决定"结尾 |
| **API Key 保护** | .env 环境变量 | 代码中零硬编码密钥，通过 .env 文件加载，.env 已加入 .gitignore |
| **数据隐私** | 用户画像本地存储 | 不传输用户健康数据到第三方服务器，仅存储于本地 |
| **危险操作确认** | 人机协同机制 | 所有涉及健康建议的输出均需用户确认，不自动执行任何不可逆操作 |

**任务描述**：在选题阶段，用户对做什么方向感到迷茫，Claude Code 没有直接列出方案清单，而是通过 10 个层层递进的提问帮助用户自己挖掘出真正的需求。

- 最终通过 10 轮提问，锁定"医疗健康智能助手"方向

**采纳结果**：整个选题过程采纳了 AI 的苏格拉底式引导方法。但初始方案"药企/器械代表助手"经 AI 搜索数据后调整为覆盖面更大的"C 端医疗消费决策"——这个调整基于真实搜索数据，是人工和 AI 共同碰撞出的结果。

**采纳结果**：用户的 3 处补充意见全部被采纳并落实到架构设计中。这个交互过程体现了人机协同的典型模式——AI 提供框架和初稿，人类基于领域知识（课程评分标准）进行补位。

**采纳结果**：报告初稿的整体框架和大部分内容被采纳。第 5 节（工程实现）的项目结构、第 7 节（测试与评估）的测试用例设计在实际编码后可能需要根据代码实况微调。

#### 样例 4：编码阶段的 Bug 排查协作

**任务描述**：在实现 Streamlit UI 时，点击"开始分析"按钮后页面刷新导致 Agent 被重复初始化，产生死循环问题。

**典型交互**：

### 6.5 对 AI 协作效果的反思

**效率提升**：在选题调研阶段，AI 代替了大量手动搜索工作（搜索趋势数据、竞争分析），将原本可能需要 2-3 天的选题过程压缩到半天。在架构设计阶段，AI 快速输出了可讨论的框架，大幅减少了"从空白开始"的启动成本。

**误导风险**：AI 偶尔会在细节上过度自信。例如初版架构中反思模块的职责描述较模糊（"检查和修正输出"），如果没有人工补充具体检查项清单，评分时该模块可能被视为"空架子"。这提醒我：AI 善长生成"面"，但"点"的深度需要人来补。

**调试成本**：编写报告时 AI 的表现很好——它本质上是结构化文本的组织和呈现。在编码阶段，AI 生成的代码经过本地运行环境下的反复调试和多轮迭代优化，最终通过了 5 个单元测试和 6 个集成测试的验证。全流程中 AI 在定位 bug、优化 prompt 边界、统一同步/异步路径等方面提供了有效的辅助。

**上下文管理经验**：整个选题到架构设计过程跨越了较长的对话，AI 始终能准确引用之前的讨论结论（如用户画像、评分标准、技术选型），未出现上下文丢失的情况。这说明 Claude Code 的上下文管理能力足以支撑一个完整项目的端到端协作。

**总结**：AI 编程智能体不适合"丢一个需求等结果"的模式。最有效的协作方式是人掌握方向（领域知识、评分标准、用户真实需求），AI 负责执行（搜索、生成、组织、检查）。本项目产出的每一个关键决策都是人机碰撞的结果，而非单方面生成。

## 8. 安全、伦理与局限性

### 8.1 敏感信息保护

- 用户健康画像数据存储于本地，不传输到任何第三方服务器
- API 调用时仅传输必要的检索文本，不附带用户身份信息
- `.env` 文件管理 API Key，已加入 `.gitignore`，代码仓库中零密钥泄露
- 提供 `.env.example` 模板文件，不包含任何真实密钥

### 8.2 危险操作确认

- 不提供诊断结论、治疗方案或药物剂量建议
- 所有输出均标注"基于公开资料生成，仅供参考，请结合医生建议做最终决定"
- 安全检测 Agent 在输入层直接拦截危险请求（自伤/伤他/紧急医疗求助）
- 关键健康建议采用人机协同模式，需用户明确确认

### 8.3 数据隐私

- 用户画像在点击"保存画像"后持久化到本地文件，不会自动上传
- 对话历史可通过界面"清除对话历史"按钮一键清除，画像数据可通过删除 data/user_profiles/ 目录手动清除
- 不上传用户原始健康报告到云端，解析过程在本地完成

### 8.4 偏见与幻觉风险

| 风险类型 | 具体表现 | 缓解措施 |
|---------|---------|---------|
| **知识库偏见** | 知识库中某类套餐/药品信息密度不均，导致推荐偏向 | 对比表显式标注信息完整度，让用户感知可用信息的差异 |
| **模型幻觉** | LLM 可能编造不存在的药品、指标或关联 | RAG 检索 + 一致性检查，输出必须与知识库原文一一对应 |
| **确认偏误** | 用户倾向于相信 AI 输出，可能影响其实际就医行为 | 全程"仅供参考"标注，关键建议强调"请咨询医生" |

### 8.5 使用边界

- **本系统适用于**：健康消费决策辅助（选体检、比药品、看懂报告）
- **本系统不适用于**：疾病诊断、急性症状处理、急诊判断、替代医生诊疗
- **免责声明**：本系统为课程作业项目，非商业医疗产品。所有信息仅供参考，不构成医疗建议

### 8.6 伦理考量

- 不利用用户健康画像进行精准营销或广告推送
- 对比结果不因商业利益偏袒特定产品或品牌
- 报告解读中不制造不必要的健康焦虑（"可能提示肝功能异常"而非"你的肝可能出了问题"）
- 尊重用户的医疗自主权——我们是辅助决策的工具，不是替用户做决定

## 9. 项目总结与未来改进

### 9.1 项目总结

MedChoice 从"智能医学工程学生能做什么"出发，经过数据驱动的选题调研和苏格拉底式需求挖掘，最终定位于"医疗健康消费决策助手"这一真实存在且有付费意愿的市场空白。

在智能体设计层面，系统构建了 4 个独立 Agent（安全检测、意图路由、场景业务、反思校验）的多智能体协作体系，完整实现了"用户输入 → 安全检测 → 意图路由 → 信息补全 → RAG 检索 → 对比分析 → 反思校验 → 人机协同输出"的 8 步工作流闭环。在课程设计模式层面，覆盖了路由、任务规划、工具使用、反思自省、记忆机制、安全护栏、人机协同 7 种模式及多智能体协作。

在工程实现层面，系统以 Python 为核心技术栈，集成了 ChromaDB 向量数据库、BGE 嵌入模型、PyMuPDF 文档解析和 LLM API，形成了可运行、可演示的完整工程。

在 AI 编程协作层面，本项目全程使用 Claude Code 作为协作工具，深刻体会到"人机协同"不仅是课程中的设计模式，也是真实的开发方法论。

### 10.2 关键配置示例（.env.example）

```bash
# LLM API 配置
LLM_PROVIDER=deepseek          # 可选: deepseek / zhipu

# DeepSeek API
DEEPSEEK_API_KEY=sk-your-deepseek-key-here
DEEPSEEK_BASE_URL=https://api.deepseek.com
DEEPSEEK_MODEL=deepseek-chat

# Zhipu API (GLM)
ZHIPU_API_KEY=your-zhipu-key-here
ZHIPU_BASE_URL=https://open.bigmodel.cn/api/paas/v4
ZHIPU_MODEL=glm-4.6v

# RAG 模式: keyword（关键词匹配，零依赖）或 embedding（BGE 语义检索）
RAG_MODE=keyword

# 嵌入模型配置（RAG_MODE=embedding 时生效，通过 ModelScope 下载）
EMBEDDING_MODEL=BAAI/bge-small-zh-v1.5

# 向量数据库路径
CHROMA_DB_PATH=./data/chroma_db

# 用户数据存储路径
USER_DATA_PATH=./data/user_profiles
```
